# TestMail Hub 产品与研发计划书

更新日期：2026-03-30

## 1. 文档目标

这份计划书用于统一项目后续的产品定位、研发主线、阶段目标和交付节奏。

后续开发默认遵循这份文档：

- 优先做高价值、可成交、可复用的能力
- 避免继续往“公开临时邮箱站”方向扩张
- 持续把项目升级为“团队测试邮箱与验证码中台”

## 2. 产品定位

### 2.1 推荐对外定位

统一建议改成：

- 团队测试邮箱与验证码中台
- 私有部署邮件资产管理后台
- 自动化测试邮件平台
- 可私有部署的邮件验证码中台

### 2.2 不建议继续主打

- 公开临时邮箱
- 免费收码工具
- 批量注册辅助平台

### 2.3 一句话定义

`TestMail Hub` 是一套面向测试团队、自动化团队和私有部署场景的邮件接收、验证码提取、邮箱资产管理、规则处理、Webhook 通知、项目隔离和发信协作平台。

## 3. 当前基线快照

截至 2026-03-30，项目已经具备下面这些核心基线：

### 3.1 已经落地的能力

- 邮件接收、正文存储、附件元数据与详情查看
- 验证码、链接、平台邮件识别
- 规则、白名单、全局白名单开关
- 邮箱资产管理与 Cloudflare 路由同步
- 项目 / 环境 / 邮箱池三层隔离
- 项目级管理员、项目级 API Token、项目级 Webhook
- Webhook 签名、投递记录、自动重试、手动重放
- 多域名资产中心第一版
- Catch-all 策略管理
- 域名监控卡片、排行图表、接入概览
- Resend 发信中心、模板、联系人、发送记录
- 管理员 Session、审计日志、系统日志
- React + TypeScript + Vite 后台
- Cloudflare Workers + D1 + GitHub Actions 发布链路

### 3.2 当前主要短板

- 验证码 / 链接提取准确率仍有优化空间
- 多域名还缺 routing profile 独立模型与 provider 抽象
- 团队治理还缺更细角色与成员体系
- 生命周期策略还缺 retention 配置中心
- 发信中心还缺更强的队列化与审批能力
- 对外销售层、演示层和 Demo 资产还没做

## 4. 北极星目标

未来 3 到 6 个月，项目的核心目标不是“更多个人用户”，而是以下三件事：

1. 做成一个稳定可用的团队测试邮箱平台。
2. 做成一个能支持私有部署和定制交付的产品底座。
3. 做成一个可以承接第一批私有部署、改造和定制项目的交付系统。

## 5. 目标客户与使用场景

### 5.1 优先客户

- 测试团队
- 自动化测试团队
- 小型研发团队
- 工作室
- 有大量验证码 / 登录邮件场景的项目组
- 需要私有部署邮件后台的客户

### 5.2 核心场景

- 自动化测试中统一接收注册、登录、验证邮件
- 从邮件中提取验证码、确认链接、魔法链接
- 用项目和环境隔离不同测试流量
- 用 API / Webhook 接入自动化流程
- 用后台统一管理邮箱、域名、规则、日志和发信

### 5.3 暂不优先场景

- 面向大众用户的公开临时邮箱网站
- 强运营导流型工具站
- 低价大众 SaaS 市场

## 6. 产品原则

### 6.1 价值优先

优先做能直接提高成交率和复用率的能力：

- 提取
- 管理
- 自动化
- 私有部署

### 6.2 风险可控

降低灰产联想，强化企业化表达：

- 少讲“临时邮箱”
- 少讲“批量注册”
- 多讲“测试”“自动化”“团队”“私有部署”
- 强化审计、权限、留痕

### 6.3 平台化优先

优先做可扩展的平台能力，而不是一次性功能：

- 项目 / 环境
- API 权限
- 事件体系
- 多域名资产
- 生命周期策略

### 6.4 可交付优先

每个阶段结束时，都应该能形成可交付成果：

- 私有部署版
- 定制版
- 自动化接入版
- 团队协作版

## 7. 阶段路线图

## 7.1 Phase 0：稳定化与产品基线

状态：已完成

目标：

- 稳定代码结构和部署链路
- 固化 CI/CD、文档、Secrets 管理
- 统一产品命名与对外口径

交付结果：

- README、CI/CD、Secrets 文档
- GitHub Actions 工作流
- 本地开发、构建、部署基线

## 7.2 Phase 1：验证码与链接提取升级

状态：大部分完成

目标：

- 从“收邮件”升级为“提取结果”
- 让系统具备真正中台价值

已实现重点：

- 验证码提取
- 登录 / 验证 / 重置 / 魔法链接提取
- 常见平台识别
- 详情页提取结果展示
- API / Webhook 返回提取结果

后续还要补：

- 更高准确率
- 提取置信度
- 误判回看
- 更多样本场景

## 7.3 Phase 2：项目 / 环境隔离

状态：核心能力已落地

目标：

- 从单后台工具升级为团队中台

已实现重点：

- 项目、环境、邮箱池三层模型
- 项目级管理员绑定
- 项目级 API Token / Webhook
- 邮件、邮箱、通知等按项目范围过滤

后续还要补：

- 更完整的团队成员模型
- 更强的工作空间治理界面

## 7.4 Phase 3：API 产品化

状态：核心能力已落地

目标：

- 让系统能直接接入自动化测试和外部系统

已实现重点：

- 最新邮件接口
- 验证码接口
- 邮件详情接口
- 附件接口
- 提取结果接口
- 托管 Token 与 project-scoped token

后续还要补：

- 更完整的 API 使用示例
- 更细的错误码与对外接入手册

## 7.5 Phase 4：Webhook 增强

状态：大部分完成

目标：

- 让系统成为自动化流程里的事件源

已实现重点：

- 事件投递
- 签名校验
- 投递记录
- 自动重试
- 手动重放
- 项目级 Webhook 范围

后续还要补：

- 更统一的事件命名体系
- 更细粒度的 attempt 明细
- 死信箱 / 死信治理策略

## 7.6 Phase 5：多域名资产中心

状态：进行中

目标：

- 从单域名工具升级为域名资产平台

已实现重点：

- 域名资产表与后台 CRUD
- `domain -> zone_id -> email_worker` 基础映射
- 域名 Catch-all 策略
- 域名与项目 / 环境绑定
- 按工作空间过滤域名池和默认域名
- 域名维度监控卡片、图表和接入概览
- 按域名做 Cloudflare 路由同步

下一步重点：

1. routing profile / catch-all 策略独立建模
2. 多 provider 抽象
3. 更细的域名级权限与运营规则

## 7.7 Phase 6：团队协作与治理

状态：部分完成

目标：

- 提升团队使用能力和企业感

已实现重点：

- 管理员体系
- Session
- 审计日志
- 系统错误日志
- 项目级管理员访问边界

下一步重点：

1. 细化角色模型
2. 团队成员管理
3. 操作备注 / 留痕增强

建议角色收敛为：

- owner
- platform_admin
- project_admin
- operator
- viewer

## 7.8 Phase 7：生命周期与策略

状态：部分完成

目标：

- 降低存储成本，提高资产可控性

已实现重点：

- 邮箱过期时间
- 到期自动停用
- 定时清理过期邮件
- 定时清理已删除邮件
- `mailbox.expired` 事件

下一步重点：

1. retention 策略表
2. 后台配置页
3. 定时任务按策略执行
4. 审计留痕

## 7.9 Phase 8：生态与商业化增强

状态：后置

目标：

- 从内部工具升级为可销售产品

重点方向：

- Playwright / Cypress 接入示例
- 演示首页 / 介绍页
- Demo 数据与演示账号
- 更强的发信中心
- 私有部署销售素材

## 8. 当前优先级排序

结合现状，建议后续严格按这个顺序推进：

1. 完成 `Phase 5 多域名资产中心`
2. 推进 `Phase 6 团队协作与治理`
3. 推进 `Phase 7 生命周期与策略`
4. 再回头增强提取准确率与对外接入体验

## 11. 对外销售表达

推荐说法：

> 这是一套团队测试邮箱与验证码管理后台。

或者：

> 这是一套可私有部署的邮件验证码中台。

核心卖点顺序：

1. 统一接收验证邮件
2. 自动提取验证码与链接
3. 管理邮箱、域名、规则和日志
4. 支持 API / Webhook / 项目隔离
5. 支持私有部署

## 12. 风险控制

需要持续注意：

- 对外文案造成的灰产联想
- API、发信能力的滥用风险
- 邮件存储与保留风险
- 权限设计过粗导致误操作
- 多域名同步时的 Cloudflare 配置复杂度

对应策略：

- 强化审计与权限
- 默认最小权限
- 生命周期策略前置
- 对发信、外部 API、自动化能力加开关
- 对外始终强调测试、自动化、团队、私有部署

## 13. 当前执行决议

从这份计划书生效起，项目默认按这条主线继续迭代：

`接收邮件 -> 提取结果 -> 管理资产 -> 自动化接入 -> 平台化治理`

如果新需求不在这条主线里，默认先评估是否延后，而不是直接插队开发。
