# TestMail Hub 产品与研发计划书

更新日期：2026-03-31

## 1. 文档目标

这份计划书用于统一项目后续的产品定位、研发主线、阶段目标和交付节奏。

后续开发默认遵循这份文档：

- 优先做高价值、可成交、可复用的能力
- 避免继续往“公开临时邮箱站”方向扩张
- 持续把项目升级为“团队测试邮箱与验证码中台”

## 2. 产品定位

### 2.1 推荐对外定位

统一建议改成：

- 团队测试邮箱与验证码中台
- 私有部署邮件资产管理后台
- 自动化测试邮件平台
- 可私有部署的邮件验证码中台

### 2.2 不建议继续主打

- 公开临时邮箱
- 免费收码工具
- 批量注册辅助平台

### 2.3 一句话定义

`TestMail Hub` 是一套面向测试团队、自动化团队和私有部署场景的邮件接收、验证码提取、邮箱资产管理、规则处理、Webhook 通知、项目隔离和发信协作平台。

## 3. 当前基线快照

截至 2026-03-31，项目已经具备下面这些核心基线：

### 3.1 已经落地的能力

- 邮件接收、正文存储、附件元数据与详情查看
- 验证码、链接、平台邮件识别
- 规则、白名单、全局白名单开关
- 邮箱资产管理与 Cloudflare 路由同步
- 项目 / 环境 / 邮箱池三层隔离
- 项目级管理员、项目级 API Token、项目级 Webhook
- Webhook 签名、投递记录、自动重试、手动重放
- 多域名资产中心第一版
- Catch-all 策略管理
- 域名监控卡片、排行图表、接入概览
- Resend 发信中心、模板、联系人、发送记录
- 管理员 Session、审计日志、系统日志
- 成员最近变更摘要、成员页内最近变更记录与审计筛选
- React + TypeScript + Vite 后台
- Cloudflare Workers + D1 + GitHub Actions 发布链路

### 3.2 当前主要短板

- 验证码 / 链接提取准确率仍有优化空间
- 多域名还缺更多 provider 落地、更细的域名治理规则与审计动作
- 团队治理还缺更细的资源级权限、成员留痕和协作体系深化
- 生命周期策略中心与执行记录已落地，下一步重点转向归档、审计留痕增强和更细粒度的清理动作
- 发信中心还缺更强的队列化与审批能力
- 对外销售层、演示层和 Demo 资产还没做

## 4. 北极星目标

未来 3 到 6 个月，项目的核心目标不是“更多个人用户”，而是以下三件事：

1. 做成一个稳定可用的团队测试邮箱平台。
2. 做成一个能支持私有部署和定制交付的产品底座。
3. 做成一个可以承接第一批私有部署、改造和定制项目的交付系统。

## 5. 目标客户与使用场景

### 5.1 优先客户

- 测试团队
- 自动化测试团队
- 小型研发团队
- 工作室
- 有大量验证码 / 登录邮件场景的项目组
- 需要私有部署邮件后台的客户

### 5.2 核心场景

- 自动化测试中统一接收注册、登录、验证邮件
- 从邮件中提取验证码、确认链接、魔法链接
- 用项目和环境隔离不同测试流量
- 用 API / Webhook 接入自动化流程
- 用后台统一管理邮箱、域名、规则、日志和发信

### 5.3 暂不优先场景

- 面向大众用户的公开临时邮箱网站
- 强运营导流型工具站
- 低价大众 SaaS 市场

## 6. 产品原则

### 6.1 价值优先

优先做能直接提高成交率和复用率的能力：

- 提取
- 管理
- 自动化
- 私有部署

### 6.2 风险可控

降低灰产联想，强化企业化表达：

- 少讲“临时邮箱”
- 少讲“批量注册”
- 多讲“测试”“自动化”“团队”“私有部署”
- 强化审计、权限、留痕

### 6.3 平台化优先

优先做可扩展的平台能力，而不是一次性功能：

- 项目 / 环境
- API 权限
- 事件体系
- 多域名资产
- 生命周期策略

### 6.4 可交付优先

每个阶段结束时，都应该能形成可交付成果：

- 私有部署版
- 定制版
- 自动化接入版
- 团队协作版

## 7. 阶段路线图

## 7.1 Phase 0：稳定化与产品基线

状态：已完成

目标：

- 稳定代码结构和部署链路
- 固化 CI/CD、文档、Secrets 管理
- 统一产品命名与对外口径

交付结果：

- README、CI/CD、Secrets 文档
- GitHub Actions 工作流
- 本地开发、构建、部署基线

## 7.2 Phase 1：验证码与链接提取升级

状态：大部分完成

目标：

- 从“收邮件”升级为“提取结果”
- 让系统具备真正中台价值

已实现重点：

- 验证码提取
- 登录 / 验证 / 重置 / 魔法链接提取
- 常见平台识别
- 详情页提取结果展示
- API / Webhook 返回提取结果

后续还要补：

- 更高准确率
- 提取置信度
- 误判回看
- 更多样本场景

## 7.3 Phase 2：项目 / 环境隔离

状态：核心能力已落地

目标：

- 从单后台工具升级为团队中台

已实现重点：

- 项目、环境、邮箱池三层模型
- 项目级管理员绑定
- 项目级 API Token / Webhook
- 邮件、邮箱、通知等按项目范围过滤

后续还要补：

- 更完整的团队成员模型
- 更强的工作空间治理界面

## 7.4 Phase 3：API 产品化

状态：核心能力已落地

目标：

- 让系统能直接接入自动化测试和外部系统

已实现重点：

- 最新邮件接口
- 验证码接口
- 邮件详情接口
- 附件接口
- 提取结果接口
- 托管 Token 与 project-scoped token

后续还要补：

- 更完整的 API 使用示例
- 更细的错误码与对外接入手册

## 7.5 Phase 4：Webhook 增强

状态：大部分完成

目标：

- 让系统成为自动化流程里的事件源

已实现重点：

- 事件投递
- 签名校验
- 投递记录
- 自动重试
- 手动重放
- attempt 级明细、响应码与耗时记录
- 死信箱、单条/批量重放、单条/批量忽略
- 通知中心汇总指标、健康状态、24h 成功率 / 耗时视图
- 事件字典统一、旧别名归一与告警摘要
- 项目级 Webhook 范围

后续还要补：

- 更统一的事件命名体系
- 更细粒度的告警、SLA 与治理审计指标

## 7.6 Phase 5：多域名资产中心

状态：进行中

目标：

- 从单域名工具升级为域名资产平台

已实现重点：

- 域名资产表与后台 CRUD
- `domain -> zone_id -> email_worker` 基础映射
- 域名 Catch-all 策略
- 独立路由策略 / Routing Profile 第一版
- provider 抽象基线与手动 / 外部托管域名第一版
- 域名治理规则第一版（允许新建邮箱 / 允许邮箱路由同步）
- 域名治理批量动作与治理审计摘要第二版
- 项目级管理员可管理本项目域名资产，默认对全局域名保持只读
- 域名与项目 / 环境绑定
- 按工作空间过滤域名池和默认域名
- 域名维度监控卡片、图表和接入概览
- 按域名做 Cloudflare 路由同步

下一步重点：

1. 更多 provider 落地
2. 更细的域名级权限与运营规则
3. 域名状态监控、审计和批量治理继续增强

## 7.7 Phase 6：团队协作与治理

状态：部分完成

目标：

- 提升团队使用能力和企业感

已实现重点：

- 管理员体系
- Session
- 审计日志
- 系统错误日志
- 项目级管理员访问边界
- 前端权限态第一轮落地：
  - 平台级页面开始区分“全局可写 / 项目级只读 / 只读角色”
  - 规则、白名单、项目空间、管理员、API Token、通知、邮箱资产页面已按权限隐藏写入口
- 前端权限态第二轮补齐：
  - 发信中心已区分发信操作权限与全局发件设置权限
  - 审计日志、系统日志已在前端直接做访问保护和受限提示
- 平台级敏感读取接口已收紧：
  - 管理员列表
  - API Token 列表
  - 审计日志
  - 系统错误日志
- 成员中心第一版已落地：
  - 支持关键词、角色、访问范围、项目、状态筛选
  - 筛选条件已下推到后端，避免只做前端假筛
  - 项目级管理员只会看到自己项目范围内的成员与项目选项
  - 成员已支持备注字段，成员变更会在审计日志中展示前后摘要
  - 成员列表已展示最近变更时间、操作人和动作
  - 成员页内可直接打开最近变更记录抽屉，不必跳到全局审计页
- 审计日志第二轮增强已落地：
  - 支持按关键词、实体类型、动作域做服务端筛选
  - 成员治理与域名治理摘要已统一为中文摘要展示

下一步重点：

1. 继续把剩余页面的前端权限态补齐
2. 在现有五档角色基础上继续细化资源级权限
3. 团队成员管理中心继续深化（资源级授权、成员资料与治理视图）
4. 操作备注 / 留痕增强

当前角色已收敛为第一版：

- owner
- platform_admin
- project_admin
- operator
- viewer

## 11. 对外销售表达

推荐说法：

> 这是一套团队测试邮箱与验证码管理后台。

或者：

> 这是一套可私有部署的邮件验证码中台。

核心卖点顺序：

1. 统一接收验证邮件
2. 自动提取验证码与链接
3. 管理邮箱、域名、规则和日志
4. 支持 API / Webhook / 项目隔离
5. 支持私有部署

## 12. 风险控制

需要持续注意：

- 对外文案造成的灰产联想
- API、发信能力的滥用风险
- 邮件存储与保留风险
- 权限设计过粗导致误操作
- 多域名同步时的 Cloudflare 配置复杂度

对应策略：

- 强化审计与权限
- 默认最小权限
- 生命周期策略前置
- 对发信、外部 API、自动化能力加开关
- 对外始终强调测试、自动化、团队、私有部署

## 13. 当前执行决议

从这份计划书生效起，项目默认按这条主线继续迭代：

`接收邮件 -> 提取结果 -> 管理资产 -> 自动化接入 -> 平台化治理`

如果新需求不在这条主线里，默认先评估是否延后，而不是直接插队开发。
